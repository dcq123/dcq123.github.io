沙箱目录

沙箱目录是一种安全数据策略，它的设计原理就是每个应用都有自己的私有目录，且只能访问自己的私有目录，不允许其他的引用程序访问。很多系统都采用了沙箱目录，像：Android和IOS 还有实现HTML5规范的一些浏览器等。使用沙箱目录后应用程序间要想实现数据共享，就需要其他的方式，如在Android中提供了ContentProvider将数据共享给其他应用，在IOS 中提供了特定的API来共享系统的数据(通讯录联系人等)。

应用程序的沙箱目录下包含Documents 、Library、tmp 这3个目录，如下：

    在Library目录下有两个目录:Caches和Preferences，其中Preferences 包含应用程序的偏好设置文件，我们不应该直接创建偏好设置文件，而是应该使用NSUserDefaults类来取得和设置应用程序的偏好；Caches 目录用来放置一些缓存数据。

对象归档

对象归档是一种序列化方式，类似于java中的Serializable ，可以将一个归档对象序列化到一个文件中，也可以使用反序列化得方式将数据恢复到对象中。它可以作为对象持久化的一种方式，但是不适合对象拥有大量数据或者数据需要频繁更新的情况。

对象归档需要满足以下条件：

  • 该对象的类必须实现NSCoding 协议
  • 该对象中的每个成员应该是基本类型或者是实现NSCoding协议的其他对象

对象归档需要用到以下3个类：

  • NSKeyedArchiver(归档类)

  • NSKeyedUnarchiver(反归档类)

  • NSData(字节数据的封装类，提供了读写数据文件的方法)

Sqlite数据库

Sqlite数据库是一种嵌入式、可移植、可靠性高的轻量级数据库，IOS和Android中都采用Sqlite 数据库。很多的ORM框架也是基于Sqlite 数据库作为基础，提供方便操作对象的关系型映射API。

IOS中操作数据的常用方法：

  • sqlite3_open() 打开数据库
  • sqlite3_close() 关闭数据库
  • sqlite3_errmsg() 获取错误信息
  • sqlite3_prepare_v2() 预处理sql语句
  • sqlite3_step() 执行语句，返回操作结果
  • sqlite3_finalize() 结束执行语句
  • sqlite3_bind_text() 绑定预编译中的文本值，不同的数据类型有类似的方法
  • sqlite3_bind_int() 绑定预编译中的Int值
  • sqlite3_column_int() 获取查询语句中指定列的Int值，不同的数据类型有类似的方法
  • sqlite3_column_text() 获取查询语句中指定列的文本值

一个常规的sql语句操作步骤：

  1. 定义sql 语句
  2. 使用sqlite3_prepare_v2()对Sql 语句进行预处理
  3. 使用sqlite3_bind_text()绑定预编译的值
  4. 使用sqlite3_step()执行语句并返回结果
  5. sqlite3_finalize()完成语句的执行

Android Drawable源码分析和自定义

Drawable简介

Drawable类似于View，它能够在Canvas 上绘制任何你想绘制的东西，但是它比View更加简单，它不用去考虑measure、layout 等过程，它不能与用户交互不能处理触摸事件，它只需要去考虑如何去draw就行了。Drawable是一个抽象类，在日常开发中我们大多情况下并不是直接使用Drawable，我们用的更多的是在xml标签 ，比如selector、shape、layer-list等等，这些xml标签 并不是单独存在的，它们每一个都对应着一个Drawable的实现类。

Android通用的混淆配置

Android 程序开发中，在打release包时一般都会对APP进行混淆，一方面是为了压缩和优化代码，另一方面也是为了给反编译添加一层保障；当然并不是说只有release包可以混淆，debug包也是可以混淆的，只是混淆后debug包断点无法正常调试。

本文搜集并结合自己的理解，将混淆的使用以及注意事项加以总结，最后提供一个通用的混淆模板，方便自己查看和使用。

Proguard介绍

ProGuard是一个开源的Java代码混淆器，由Google提供并集成在Android的tool工具中，它可以对项目中指定的Java代码 进行混淆，但只是对Java代码 的混淆，它无法混淆native代码，资源文件drawable、xml等，对于资源的混淆可以查看下面的微信资源混淆库。

Proguard的作用

  • 压缩：移除无效的类、属性、方法等；
  • 优化：优化字节码，并删除无效的结构；
  • 混淆：将类名，属性名，方法名混淆为难以读懂的字母，提高了反编译并阅读源码的难度；

Android并不常用但是很强大的API汇总

本文主要用来总结一些开发中不常用，或者并不为人所知的一些强大API。对这些这些Cool的API进行汇总，随时待续添加内容。

startActivities()方法

其实我们绝大多数的开发者可能是没有用过这个方法的，根据我个人理解，用的到场景并不多。这个方法最直接的理解就是使用intent开启多个Activity，我在Google的关于Activity.startActivities()文档说明中，并没有获取到除了StartActivity之外更多的信息。于是我继续扒源码，果然在Context和ContextCompat 下找到了更加详细的说明和解释。

Context，以及 documentation of ContextCompat 

* Launch multiple new activities.  This is generally the same as calling
* {@link #startActivity(Intent)} for the first Intent in the array,
* that activity during its creation calling {@link #startActivity(Intent)}
* for the second entry, etc.  Note that unlike that approach, generally
* none of the activities except the last in the array will be created
* at this point, but rather will be created when the user first visits
* them (due to pressing back from the activity on top).

从上面两处文档解释我们可以理解到的，startActiviyies 会创建一个新的Task Stack，任务栈里Activity的位置基于我们传入的Intent数组。当我们按返回键时，就会将栈顶的Activity移除。。其实这也是framework管理用户开启的activity的方式。这里结合Activity的启动模式来理解，就简单多了。

至于应用场景，我目前能想到的就是点击通知栏来开启应用的Activities、APP启动图点击后的跳转逻辑等，而开启哪些Activity以及相应顺序，我们就可以用到这个方法了。按照顺序最后一个Activity展示在栈顶，当按返回键时，一级一级返回到最前面的Activity。

ViewPager使用需要注意的细节

ViewPager是Support包中提供的类，使用也是相当广泛了，但是在使用中不同的场景可能需要不同的效果，此时就需要对ViewPager内部做些简单了解，了解其中一些可能变成坑的地方。

默认多加载一页

ViewPager在直接使用时默认会帮助我们加载下一页的界面及数据，当滚动到中间时，会保留前一个和后一个页面不被销毁，但是有些情况下我们可能不想让ViewPager去加载下一页页面及数据，例如一些新闻APP，大多就是使用ViewPager+Fragment，当用户没有滑动到下一页就自动去加载下一页数据时显然会耗费用户流量，此时就需要修改ViewPage的这种默认加载。

查看ViewPager代码会发现其定义了这个常量：

private static final int DEFAULT_OFFSCREEN_PAGES = 1;

这个常量就是预加载的页面数量，可以直接根据需要将其设置为0，禁止默认加载。

ps:由于ViewPager类在Support包中是一个独立的类，所以我们完全可以将其拿出来，对其进行定制修改，只需要全部复制，更换个包名即可放到我们自己的项目中使用。这样也就可以随意修改ViewPager的源码了。

ViewPager不动态刷新的问题

ViewPager不能动态刷新，主要是因为其中的PagerAdapter的notifyDataSetChanged()方法会失效。其原因就是当ViewPager绘制完Item之后，ViewPager会将child标记成POSITION_UNCHANGED，这样就不会在notifyDataSetChanged调用时去更新这个Item中的View了，所以界面这个方法是重新PagerAdapter的getItemPosition()方法。
