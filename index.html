Android高斯模糊的实现方案

高斯模糊（英语：Gaussian Blur）是一种图片处理技术，很多App中背景图片使用高斯模糊处理后，视觉效果更佳。实现高斯模糊的方式也有很多，下面会简单针对常用的实现方案进行比较。

常用的实现方式：

  1. Java算法实现
  2. Android 提供的RenderScript实现
  3. NDK实现的算法
  4. openGL处理

Github有个项目将不同的blur实现进行比较：BlurTestAndroid
借用上面的一张图进比较：

从图中可以清晰的看到比较：

  • RS_GAUSS_FAST是使用Renderscript框架中提供的ScriptIntrinsicBlur 实现的，它是在Android中最快最好的高斯模糊算法。
  • RS_BOX_5x5,RS_GAUSS_5x5 也是用由Renderscript框架提供基于卷积矩阵实现高斯模糊，它是使用类ScriptIntrinsicConvolve来实现。
  • STACKBLUR 是一个用Java实现的高斯模糊算法(BlurEffectForAndroidDesign)(该Lib 在Android API > 16时使用的也是RenderScript)
  • RS_STACKBLUR 也是一个用Renderscript 实现的libandroid-stackblur
  • GAUSS_FAST 是一个用Java方法实现.算法地址
  • BOX_BLUR 一个用Java方法实现算法地址

从上面的比较可以看出，使用ScriptIntrinsicBlur 实现是最高效的，但是ScriptIntrinsicBlur API 从Android 17才开始提供,也就是Adnroid 4.2以下版本都无法使用，幸运的是Google为我们提供了Support包，只需要简单配置，在低版本的Android设备上也可以使用ScriptIntrinsicBlur。

Gradle打正常Jar和混淆Jar包

前言

近期在做一个SDK的项目，需要以Jar包的形式对外提供，项目中包含了布局，图片等资源文件，一般这种情况以aar对外提供是最方便的，可以将项目中所有的class文件和资源文件统一打包的aar中，而且是在编译后，studio自动会打包成aar放在项目的build/outputs/aar目录下，非常方便。

但是，这个项目要求，就是需要Jar包的形式对外提供。这种情况就设计到资源文件如何处理，当然，没有别的办法，如果以Jar的形式对外提供，资源文件就只能也单独拿出来，提供出去，别人在集成时，将提供的所有资源文件放置到指定的资源目录中。友盟分享就有这种集成方式，所以既然项目需要，那就也按照这种方式来吧~但是但是，资源文件这部分有大坑，后面再来说下这个坑以及解决方案。

资源文件存在的大坑

我们都知道，所有的资源文件都会自动在R文件中生成一个常量，我们在java类中使用到所有的资源都是以R.xx.xxx的形式去获取。这种情况下，如果我们在SDK的类文件中以这种形式使用资源，然后在对外以Jar包的形式提供，资源文件也单独提供，让使用者集成，当这些资源文件放入到使用者的项目中时，所有的资源id常量就又重新生成了一次，而且肯定和原来SDK中的R文件中的常量值不一样。这种情况下，如果直接使用提供的Jar包，肯定是各种资源找不到，包括原来引入的那个R文件也是木有的，所以资源单独提供这种方式，必须要解决资源引用问题。否则前面打出的jar包都是没用的！！

文件共享FileProvider的使用

随着Android 7.0的发布，系统对权限控制更加严格，从6.0开始的动态权限，到7.0对应用私有目录访问限制，都可以看到Android系统的隐私保护的力度越来越大。随着安全性的增强，开发者也需要作出跟多的处理适配最新的权限限制。本文主要解决私有目录文件共享的问题，主要方式就是通过FileProvider进行共享。

应用间共享文件

需求：应用A可以对外提供图片共享，共享的是应用A内部私有目录下的图片资源，应用B通过Intent发送获取图片的请求，从应用A中获取共享的图片，并显示到应用B内。以上场景类似与从相册选取图片并显示的应用中。

配置共享目录

上面的resourece中的xml对应的文件filepaths就是共享目录配置文件，该文件必须是xml文件，放在项目的xml目录下。

所有的共享目录都需要在<paths>内指定，而且他们是有一定的规范，像<files-path>标签并不是随意定义的xml标签。有一下几种类型的标签：

  • <files-path />

    表示应用内部存储空间中的文件子目录，即：通过Context.getFilesDir()获取到的目录

  • <cache-path />

    表示应用内部的缓存子目录，即：通过getCacheDir()获取到的目录。

  • <external-path />

    表示外部存储子目录，Environment.getExternalStorageDirectory()获取到的目录。

  • <external-files-path />

    表示外部存储目录中的文件子目录，Context#getExternalFilesDir(String)获取的目录。

  • <external-cache-path />

    表示外部存储目录中的缓存子目录，Context.getExternalCacheDir()获取到的目录。

以上的每个标签都有path和name属性，其中path用来指定对应的目录下的子路径，如果path=""则代表使用对应的目录作为根目录，即：如果使用的是<file-paths /> 那么内部的存储空间files就是共享的根目录。

临时权限设置

文件共享中的关键一步就是共享的应用(应用A)需要临时授权希望获取的应用(应用B)有访问自身私有目录下文件的权限。设置这种权限有以下两种方式：

使用grantUriPermission()

调用Context.grantUriPermission(package, Uri, mode_flags)方法对指定的uri设置授权，mode_flages可以设置为FLAG_GRANT_READ_URI_PERMISSION或 FLAG_GRANT_WRITE_URI_PERMISSION或者两个都设置。但是这种权限是一种持久权限，一旦授予后就会持续有效，直到设备重启或者手动调用revokeUriPermission()方法回收权限。

Intent中添加Flags

使用这种方式需要：1、将共享的uri设置到intent的data(调用intent的setData())；2、为intent添加flags可以为 FLAG_GRANT_READ_URI_PERMISSION or FLAG_GRANT_WRITE_URI_PERMISSIONor 两个一起；3、使用setResult() 将Intent返回。这种方式的授权，权限是在调用共享的Activity的生命周期只能，一旦该Activity出栈了，该权限就失效了。

需要说明的是，在通常的使用中都会选择第二种方式，第一种方式权限的有效期时间过长，虽然可以控制但是需要手动去调用回收权限的操作，这种情况对触发调用的时机也不好把控，第二种授予的权限的有效期随着Activity的生命周期一起结束，这种情况更加满足我们的需求。

SurfaceView

surfaceView是一种特殊的View，它可以在子线程中进行绘制，surfaceView并没有使用View的绘制窗口，它有自己独自的surface，它的所有绘制操作都是在这个独立的surface上进行的。SurfaceView中主要是通过SurfaceHolder来控制surface的(surfaceHolder是一个接口，提供了很多操作surface的方法)，其中SurfaceHolder.Callback用来管理surface的生命周期，也是使用最多的接口。

在自定义相机时，需要使用SurfaceView来呈现预览视图，同时在SurfaceHolder.Callback对应的生命周期方法中，可以对相机的预览视图进行监控。

SurfaceHolder.Callback接口
  • surfaceCreated(SurfaceHolder holder)

当Surface第一次创建时会立即调用该方法，可以在该方法中做和绘制相关的初始化操作，在自定义相机时，可以在该方法中调用setPreviewDisplay()或startPreview()方法开启预览。

  • surfaceChanged(SurfaceHolder holder, int format, int width,int height)

当Surface的状态（大小和格式）发生变化的时候会调用该函数，在surfaceCreated调用后该函数至少会被调用一次。 自定义相机时，可以在此处stopPreview()并重新调用开启预览方法。

  • surfaceDestroyed(SurfaceHolder holder)

当Surface被摧毁前会调用该函数，该函数被调用后就不能继续使用Surface了，一般在该函数中来清理使用的资源。 自定义相机时，可以在此进行资源释放的操作。

需要注意的问题

以下针对在自定义相机时遇到的问题进行总结说明

预览画面中的图片被拉伸变形

该问题是因为SurfaceView和PreviewSize的长宽比不一致导致的，因为手机预览视图是由相机预览图像根据SurfaceView的大小缩放得来的，当长宽比不一致时必然后导致图像变形。

拍摄照片中物体被拉伸变形

这个问题是Previewsize和Picturesize的长宽比率不一致所致，具体问题是跟某些相机硬件底层有关。

PS: 为了确保预览视图及获取到的图片正常不被变形，在开发时最好将SurfaceView、PreviewSize、PictureSize三个尺寸保证长宽比例一致。具体实现可以先通过camera.getSupportedPreviewSizes()和camera.getSupportedPictureSizes()获得相机硬件支持的所有预览和拍摄尺寸，然后在里面筛选出和SurfaceView的长宽比一致并且大小合适的尺寸，通过camera.setPrameters来更新设置。
注意：市场上手机相机硬件支持的尺寸一般都是主流的4:3或者16:9，所以SurfaceView尺寸不能太奇葩，最好也设置成这样的长宽比。

前置摄像头镜像问题

Android相机硬件有个特殊设定，就是对于前置摄像头，在展示预览视图时采用类似镜面的效果，显示的是摄像头成像的镜像。而拍摄出的照片则仍采用摄像头成像。这是由于底层相机在传递前置摄像头预览数据时做了水平翻转变换，即将x方向镜像翻转180度。这个变化对之前竖屏预览的方向也会造成影响，本来对于后置摄像头旋转90度即可使预览视图正确，而对前置摄像头，如果也旋转90度的话，看到的预览图像则是上下颠倒的（因为x方向翻转了180度），因此必须再旋转180度，才能显示正确。

前置摄像头的预览方向示意图：

此外，由于拍摄图像并没有做水平翻转，所以对于前置摄像头拍出来的照片，用户会发现跟预览时所见的是左右翻转的。这个在一定程度上会影响用户体验。为了解决这个问题，可以对前置摄像头拍摄的图像在生成位图文件时增加一个水平翻转矩阵变换。

开始之前需要先考虑一个问题

设计师在出图时，肯定会标注二维码，标题，商品图片在整个合成图中的位置(其实也就是在背景底部上的位置)，但是背景底图是从服务端获取的，并不是存在本地的drawble-xxx目录下，所以这里就出现了一个适配的问题。

方案一：服务端存放两套底图，一套是xhdpi的，一套是xxhdpi的(一般我们做在适配时，没有特别需要处理的，出这两套图基本可以很好的适配大部分机型了。)客户端在请求服务器时，先判断当前设备的屏幕分辨率，取接近720p 的作为与xhdpi 对应，取接近1080p的和xxhdpi对应，告知服务器，当前设备需要请求哪套图片，根据这种方式做适配。

但是这种方式经尝试后，并没有很好的生成要求的合成图。因为设计标出的尺寸单位(dp)，在合成图片时，使用的都是px，所以需要将所有设计图上的dp尺寸转成px进行位置放置，这时对于高分辨率的(能达到xxxhdpi)，显然两套图就不能适配了。而且更高分辨率的底图也会更大，获取底图的时间也会更长，图片合成时占用的内存也就更多。而且合成的二维码海报也并不是为了在APP内展示，只是为了保存至本地，并能进行分享。

所以决定不使用这种方式，就使用一套图，就使用px作为尺寸单位。

方案二： 服务端只有一套图xxhdpi尺寸的，设计师标注的尺寸也是在xxhdpi上进行标注的，而且单位是px，这样所有合成时需要使用的尺寸都直接写px即可，也不用去判断当时设备需要适配那种分辨率的尺寸。因为这张合成图，压根就不展示。所以直接用固定px去合成就能满足需求。

一般根据依赖进行相应变化的逻辑就在onDependentViewChanged()中去实现。

onDependentViewChanged

该方法就是用来响应dependent view 的变化的，无论何时dependent view的大小、位置发生变化时都会调用该方法。Behavior 使用该方法在合适的时候去更新child view 的展示。

onDependentViewRemoved

该方法是在child 所依赖的dependent 被remove后调用，我们可以在这个方法中针对dependent 移除后更新child 的显示。

示例

下面通过FloatingActionButton.Behavior的分析，对以上3个方法的使用做个简单介绍。

不设置依赖，根据触摸事件做出相应变化

正如前面提到的Behavior就像一个View一样，它可以拦截并处理触摸事件，提供这个机会的就是CoordinatorLayout，它作为父布局，在接收到触摸事件后，会将该触摸事件传递给Behavior的onInterceptTouchEvent()，让Behavior有一次拦截触摸事件的机会。如果onInterceptTouchEvent()方法返回true，就会回调Behavior的onTouchEvent()方法出处理所有的触摸事件，而对应的View完全不知道该事件的存在，所以一旦事件交由Behavior处理了，View就不会再收到触摸事件了。

PS:在处理触摸事件的场景，可以不用设置依赖，事件会正常由CoordinatorLayout传递到Behavior中。

以下针对触摸事件处理的场景下，涉及到Behavior中的方法进行介绍：

onInterceptTouchEvent

该方法是在CoordinatorLayout为子View分发触摸事件之前调用的方法，该方法内部可以根据一些条件判断Behavior是否要处理相应的触摸事件，默认情况下该方法返回false，表示Behavior不处理任何触摸事件。如果想在某种条件下想让Behavior处理触摸事件，返回true即可，一旦返回为true后，CoordinatorLayout后续收到的所有触摸事件都会传递到Behaviro的onTouchEvent()方法中，由该方法来响应全部的触摸事件，别的View将不会再收到任何事件。

onTouchEvent

该方法和view中的onTouchEvent()方法一样，都是用来处理具体的触摸事件的。可以在该方法中根据对应的触摸事件操作CoordinatorLayout中子view的显示行为。这个方法应该去执行真实的view的布局变化的逻辑。

blocksInteractionBelow

该方法也可以用来处理触摸事件拦截，从方法名可以看出一旦该方法返回true，它会阻断该视图下其他视图的所有交互事件，也就是说该视图下的其他视图将无法再收到任何触摸事件，该方法的返回值取决于getScrimOpacity()方法，这个方法是在事件被阻断的情况下，系统希望在该情况下能有一些视觉效果的变化，告知用户底部的视图是无法再响应事件的，getScrimOpacity()方法返回一个非零将在View之上绘制一层overlay颜色并且屏蔽所有的交互。

getScrimOpacity

该方法默认返回0，所以也就代表blocksInteractionBelow()方法默认返回false，返回大于0的值时，就表示当阻断了当前视图下面的所有子视图的交互时，存在一个类似与Dialog的遮罩层，能够在视觉上告知用户，后面的是视图是不能够交互的。我们可以使用getScrimColor()方法指定遮罩层的颜色。

getScrimColor

该方法指定遮罩层的颜色，返回的是一个int类型的颜色值，默认返回黑色。

CoordinatorLayout的基本使用

官方注释：

CoordinatorLayout is a super-powered FrameLayout.

CoordinatorLayout is intended for two primary use cases:

1、As a top-level application decor or chrome layout
2、As a container for a specific interaction with one or more child views

简单的说，单从布局上来看，CoordinatorLayout是一个增强型的FrameLayout(它继承ViewGroup但是它内部的布局行为和FragmentLayout是类似的，所以我们可以在布局是时把它当做一个FragmentLayout，可以使用所有FragmentLayout能用的布局属性。)，它是一个用来协调各个子View之间交互动作的容器View。而Behavior就是用来给CoordinatorLayout的子view们实现交互各自的交互行为的。Behavior 是CoordinatorLayout的核心。

  • exitUntilCollapsed

    滚动退出屏幕，最后折叠在顶端。（上面的示例效果就是该属性产生的）

AppBarLayout

AppBarLayout 是继承LinerLayout实现的一个ViewGroup容器组件，支持手势滑动操作。默认的AppBarLayout是垂直方向的，它的作用是把AppBarLayout包裹的内容都作为AppBar，作为一个整体去控制。AppBarLayout必须作为Toolbar的父布局容器才能使Toolbar具备一些行为和效果。

CollapsingToolbarLayout

CollapsingToolbarLayout包裹 Toolbar 的时候提供一个可折叠的 Toolbar，一般作为AppbarLayout的子视图使用，它就是一个常规的布局，并没有实现相应的Behavior。

CollapsingToolbarLayout提供了以下属性：

  • title：设置ToolBar的标题，当CollapsingToolbarLayout全屏没有折叠时，title显示的是大字体，在折叠的过程中，title不断变小到一定大小的效果。你可以调用setTitle(CharSequence)方法设置title。当设置该属性后，会覆盖Toolbar设置的title。
  • titleEnabled：如果将其设置为false，那么title就始终显示在顶部的位置，也不会有字体从大到小的过渡和移动，默认是true。
  • expandedTitleMargin：设置在全屏未折叠时Title对应的间距，类似与这样的属性还有多个，它们都是只在未折叠时的设置，而且需要确保为折叠时CollapsingToolbarLayout所占据的空间足够大，否则会有被挤出显示范围的情况。
  • contentScrim：ToolBar被折叠到顶部固定时候的背景，你可以调用setContentScrim(Drawable)方法改变背景或者 在属性中使用 app:contentScrim=”?attr/colorPrimary”来改变背景。
  • layout_collapseMode：子视图的折叠模式，它有以下两个值：
    • pin ： 固定模式，在折叠的时候最后固定在顶端。
    • parallax : 视差模式，在折叠的时候会有个视差折叠的效果
