CoordinatorLayout Behavior详细介绍

Behavior是整个CoordinatorLayout的核心，它就向一个ViewGroup一样，不但可以测量和布局子View，而且可以拦截和处理事件，并且能接受和处理嵌套滚动事件。它在CoordinatorLayout中是一个抽象的存在，无论在布局、测量，还是在处理基本事件或是嵌套滚动时，都会去调用其子View中设置了Behavior的具体实现，无论该View是CoordinatorLayout的直接子view，还是子View中包含的View，只要设置了Behaivior，就都有机会接收到CoordinatorLayout的回调处理。Design Library大量利用了Behaviors来实现你所看到的功能。

设置依赖和依赖改变时回调的方法
layoutDependsOn

该方法用来决定是否将指定的某个view作为其他同级姊妹的布局依赖。

按照官方文档说明，该方法在请求布局时至少会被调用一次。如果在给定的child view和dependency view之间确定了依赖关系返回了true，父布局CoordinatorLayout会：

  • 在布局时，不管child的顺序如何，CoordinatorLayout 始终会在完成dependent view后再对child view 进行布局。
  • 当dependency view的布局或位置发生变化后，将调用onDependentViewChanged（） 方法。

一般根据依赖进行相应变化的逻辑就在onDependentViewChanged()中去实现。

onDependentViewChanged

该方法就是用来响应dependent view 的变化的，无论何时dependent view的大小、位置发生变化时都会调用该方法。Behavior 使用该方法在合适的时候去更新child view 的展示。

onDependentViewRemoved

该方法是在child 所依赖的dependent 被remove后调用，我们可以在这个方法中针对dependent 移除后更新child 的显示。

不设置依赖，根据触摸事件做出相应变化

正如前面提到的Behavior就像一个View一样，它可以拦截并处理触摸事件，提供这个机会的就是CoordinatorLayout，它作为父布局，在接收到触摸事件后，会将该触摸事件传递给Behavior的onInterceptTouchEvent()，让Behavior有一次拦截触摸事件的机会。如果onInterceptTouchEvent()方法返回true，就会回调Behavior的onTouchEvent()方法出处理所有的触摸事件，而对应的View完全不知道该事件的存在，所以一旦事件交由Behavior处理了，View就不会再收到触摸事件了。

PS:在处理触摸事件的场景，可以不用设置依赖，事件会正常由CoordinatorLayout传递到Behavior中。

以下针对触摸事件处理的场景下，涉及到Behavior中的方法进行介绍：

onInterceptTouchEvent

该方法是在CoordinatorLayout为子View分发触摸事件之前调用的方法，该方法内部可以根据一些条件判断Behavior是否要处理相应的触摸事件，默认情况下该方法返回false，表示Behavior不处理任何触摸事件。如果想在某种条件下想让Behavior处理触摸事件，返回true即可，一旦返回为true后，CoordinatorLayout后续收到的所有触摸事件都会传递到Behaviro的onTouchEvent()方法中，由该方法来响应全部的触摸事件，别的View将不会再收到任何事件。

onTouchEvent

该方法和view中的onTouchEvent()方法一样，都是用来处理具体的触摸事件的。可以在该方法中根据对应的触摸事件操作CoordinatorLayout中子view的显示行为。这个方法应该去执行真实的view的布局变化的逻辑。

blocksInteractionBelow

该方法也可以用来处理触摸事件拦截，从方法名可以看出一旦该方法返回true，它会阻断该视图下其他视图的所有交互事件，也就是说该视图下的其他视图将无法再收到任何触摸事件，该方法的返回值取决于getScrimOpacity()方法，这个方法是在事件被阻断的情况下，系统希望在该情况下能有一些视觉效果的变化，告知用户底部的视图是无法再响应事件的，getScrimOpacity()方法返回一个非零将在View之上绘制一层overlay颜色并且屏蔽所有的交互。

getScrimOpacity

该方法默认返回0，所以也就代表blocksInteractionBelow()方法默认返回false，返回大于0的值时，就表示当阻断了当前视图下面的所有子视图的交互时，存在一个类似与Dialog的遮罩层，能够在视觉上告知用户，后面的是视图是不能够交互的。我们可以使用getScrimColor()方法指定遮罩层的颜色。

getScrimColor

该方法指定遮罩层的颜色，返回的是一个int类型的颜色值，默认返回黑色。

CoordinatorLayout的基本使用

官方注释：

CoordinatorLayout is a super-powered FrameLayout.

CoordinatorLayout is intended for two primary use cases:

1、As a top-level application decor or chrome layout
2、As a container for a specific interaction with one or more child views

简单的说，单从布局上来看，CoordinatorLayout是一个增强型的FrameLayout(它继承ViewGroup但是它内部的布局行为和FragmentLayout是类似的，所以我们可以在布局是时把它当做一个FragmentLayout，可以使用所有FragmentLayout能用的布局属性。)，它是一个用来协调各个子View之间交互动作的容器View。而Behavior就是用来给CoordinatorLayout的子view们实现交互各自的交互行为的。Behavior 是CoordinatorLayout的核心。

  • exitUntilCollapsed

    滚动退出屏幕，最后折叠在顶端。（上面的示例效果就是该属性产生的）

AppBarLayout

AppBarLayout 是继承LinerLayout实现的一个ViewGroup容器组件，支持手势滑动操作。默认的AppBarLayout是垂直方向的，它的作用是把AppBarLayout包裹的内容都作为AppBar，作为一个整体去控制。AppBarLayout必须作为Toolbar的父布局容器才能使Toolbar具备一些行为和效果。

CollapsingToolbarLayout

CollapsingToolbarLayout包裹 Toolbar 的时候提供一个可折叠的 Toolbar，一般作为AppbarLayout的子视图使用，它就是一个常规的布局，并没有实现相应的Behavior。

CollapsingToolbarLayout提供了以下属性：

  • title：设置ToolBar的标题，当CollapsingToolbarLayout全屏没有折叠时，title显示的是大字体，在折叠的过程中，title不断变小到一定大小的效果。你可以调用setTitle(CharSequence)方法设置title。当设置该属性后，会覆盖Toolbar设置的title。
  • titleEnabled：如果将其设置为false，那么title就始终显示在顶部的位置，也不会有字体从大到小的过渡和移动，默认是true。
  • expandedTitleMargin：设置在全屏未折叠时Title对应的间距，类似与这样的属性还有多个，它们都是只在未折叠时的设置，而且需要确保为折叠时CollapsingToolbarLayout所占据的空间足够大，否则会有被挤出显示范围的情况。
  • contentScrim：ToolBar被折叠到顶部固定时候的背景，你可以调用setContentScrim(Drawable)方法改变背景或者 在属性中使用 app:contentScrim=”?attr/colorPrimary”来改变背景。
  • layout_collapseMode：子视图的折叠模式，它有以下两个值：
    • pin ： 固定模式，在折叠的时候最后固定在顶端。
    • parallax : 视差模式，在折叠的时候会有个视差折叠的效果

沙箱目录

沙箱目录是一种安全数据策略，它的设计原理就是每个应用都有自己的私有目录，且只能访问自己的私有目录，不允许其他的引用程序访问。很多系统都采用了沙箱目录，像：Android和IOS 还有实现HTML5规范的一些浏览器等。使用沙箱目录后应用程序间要想实现数据共享，就需要其他的方式，如在Android中提供了ContentProvider将数据共享给其他应用，在IOS 中提供了特定的API来共享系统的数据(通讯录联系人等)。

对象归档

对象归档是一种序列化方式，类似于java中的Serializable ，可以将一个归档对象序列化到一个文件中，也可以使用反序列化得方式将数据恢复到对象中。它可以作为对象持久化的一种方式，但是不适合对象拥有大量数据或者数据需要频繁更新的情况。

对象归档需要满足以下条件：

  • 该对象的类必须实现NSCoding 协议
  • 该对象中的每个成员应该是基本类型或者是实现NSCoding协议的其他对象

对象归档需要用到以下3个类：

  • NSKeyedArchiver(归档类)

  • NSKeyedUnarchiver(反归档类)

  • NSData(字节数据的封装类，提供了读写数据文件的方法)

Sqlite数据库

Sqlite数据库是一种嵌入式、可移植、可靠性高的轻量级数据库，IOS和Android中都采用Sqlite 数据库。很多的ORM框架也是基于Sqlite 数据库作为基础，提供方便操作对象的关系型映射API。

IOS中操作数据的常用方法：

  • sqlite3_open() 打开数据库
  • sqlite3_close() 关闭数据库
  • sqlite3_errmsg() 获取错误信息
  • sqlite3_prepare_v2() 预处理sql语句
  • sqlite3_step() 执行语句，返回操作结果
  • sqlite3_finalize() 结束执行语句
  • sqlite3_bind_text() 绑定预编译中的文本值，不同的数据类型有类似的方法
  • sqlite3_bind_int() 绑定预编译中的Int值
  • sqlite3_column_int() 获取查询语句中指定列的Int值，不同的数据类型有类似的方法
  • sqlite3_column_text() 获取查询语句中指定列的文本值

一个常规的sql语句操作步骤：

  1. 定义sql 语句
  2. 使用sqlite3_prepare_v2()对Sql 语句进行预处理

Drawable简介

Drawable类似于View，它能够在Canvas 上绘制任何你想绘制的东西，但是它比View更加简单，它不用去考虑measure、layout 等过程，它不能与用户交互不能处理触摸事件，它只需要去考虑如何去draw就行了。Drawable是一个抽象类，在日常开发中我们大多情况下并不是直接使用Drawable，我们用的更多的是在xml标签 ，比如selector、shape、layer-list等等，这些xml标签 并不是单独存在的，它们每一个都对应着一个Drawable的实现类。

Android通用的混淆配置

Android 程序开发中，在打release包时一般都会对APP进行混淆，一方面是为了压缩和优化代码，另一方面也是为了给反编译添加一层保障；当然并不是说只有release包可以混淆，debug包也是可以混淆的，只是混淆后debug包断点无法正常调试。

本文搜集并结合自己的理解，将混淆的使用以及注意事项加以总结，最后提供一个通用的混淆模板，方便自己查看和使用。

Proguard介绍

ProGuard是一个开源的Java代码混淆器，由Google提供并集成在Android的tool工具中，它可以对项目中指定的Java代码 进行混淆，但只是对Java代码 的混淆，它无法混淆native代码，资源文件drawable、xml等，对于资源的混淆可以查看下面的微信资源混淆库。

Proguard的作用

  • 压缩：移除无效的类、属性、方法等；
  • 优化：优化字节码，并删除无效的结构；
  • 混淆：将类名，属性名，方法名混淆为难以读懂的字母，提高了反编译并阅读源码的难度；

ViewPager使用需要注意的细节

ViewPager是Support包中提供的类，使用也是相当广泛了，但是在使用中不同的场景可能需要不同的效果，此时就需要对ViewPager内部做些简单了解，了解其中一些可能变成坑的地方。

默认多加载一页

ViewPager在直接使用时默认会帮助我们加载下一页的界面及数据，当滚动到中间时，会保留前一个和后一个页面不被销毁，但是有些情况下我们可能不想让ViewPager去加载下一页页面及数据，例如一些新闻APP，大多就是使用ViewPager+Fragment，当用户没有滑动到下一页就自动去加载下一页数据时显然会耗费用户流量，此时就需要修改ViewPage的这种默认加载。

查看ViewPager代码会发现其定义了这个常量：

private static final int DEFAULT_OFFSCREEN_PAGES = 1;

这个常量就是预加载的页面数量，可以直接根据需要将其设置为0，禁止默认加载。

ps:由于ViewPager类在Support包中是一个独立的类，所以我们完全可以将其拿出来，对其进行定制修改，只需要全部复制，更换个包名即可放到我们自己的项目中使用。这样也就可以随意修改ViewPager的源码了。

ViewPager不能动态刷新，主要是因为其中的PagerAdapter的notifyDataSetChanged()方法会失效。其原因就是当ViewPager绘制完Item之后，ViewPager会将child标记成POSITION_UNCHANGED，这样就不会在notifyDataSetChanged调用时去更新这个Item中的View了，所以界面这个方法是重新PagerAdapter的getItemPosition()方法。
