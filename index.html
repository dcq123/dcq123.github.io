  • 序列号 — 一个由 adb 创建的字符串，用于通过其控制台端口号唯一标识模拟器/设备实例。序列号的格式为 type-console-port。下面是一个序列号示例：emulator-5554

Android高斯模糊的实现方案

高斯模糊（英语：Gaussian Blur）是一种图片处理技术，很多App中背景图片使用高斯模糊处理后，视觉效果更佳。实现高斯模糊的方式也有很多，下面会简单针对常用的实现方案进行比较。

常用的实现方式：

  1. Java算法实现
  2. Android 提供的RenderScript实现
  3. NDK实现的算法
  4. openGL处理

Github有个项目将不同的blur实现进行比较：BlurTestAndroid
借用上面的一张图进比较：

从图中可以清晰的看到比较：

  • RS_GAUSS_FAST是使用Renderscript框架中提供的ScriptIntrinsicBlur 实现的，它是在Android中最快最好的高斯模糊算法。
  • RS_BOX_5x5,RS_GAUSS_5x5 也是用由Renderscript框架提供基于卷积矩阵实现高斯模糊，它是使用类ScriptIntrinsicConvolve来实现。
  • STACKBLUR 是一个用Java实现的高斯模糊算法(BlurEffectForAndroidDesign)(该Lib 在Android API > 16时使用的也是RenderScript)
  • RS_STACKBLUR 也是一个用Renderscript 实现的libandroid-stackblur
  • GAUSS_FAST 是一个用Java方法实现.算法地址
  • BOX_BLUR 一个用Java方法实现算法地址

从上面的比较可以看出，使用ScriptIntrinsicBlur 实现是最高效的，但是ScriptIntrinsicBlur API 从Android 17才开始提供,也就是Adnroid 4.2以下版本都无法使用，幸运的是Google为我们提供了Support包，只需要简单配置，在低版本的Android设备上也可以使用ScriptIntrinsicBlur。

前言

近期在做一个SDK的项目，需要以Jar包的形式对外提供，项目中包含了布局，图片等资源文件，一般这种情况以aar对外提供是最方便的，可以将项目中所有的class文件和资源文件统一打包的aar中，而且是在编译后，studio自动会打包成aar放在项目的build/outputs/aar目录下，非常方便。

但是，这个项目要求，就是需要Jar包的形式对外提供。这种情况就设计到资源文件如何处理，当然，没有别的办法，如果以Jar的形式对外提供，资源文件就只能也单独拿出来，提供出去，别人在集成时，将提供的所有资源文件放置到指定的资源目录中。友盟分享就有这种集成方式，所以既然项目需要，那就也按照这种方式来吧~但是但是，资源文件这部分有大坑，后面再来说下这个坑以及解决方案。

资源文件存在的大坑

我们都知道，所有的资源文件都会自动在R文件中生成一个常量，我们在java类中使用到所有的资源都是以R.xx.xxx的形式去获取。这种情况下，如果我们在SDK的类文件中以这种形式使用资源，然后在对外以Jar包的形式提供，资源文件也单独提供，让使用者集成，当这些资源文件放入到使用者的项目中时，所有的资源id常量就又重新生成了一次，而且肯定和原来SDK中的R文件中的常量值不一样。这种情况下，如果直接使用提供的Jar包，肯定是各种资源找不到，包括原来引入的那个R文件也是木有的，所以资源单独提供这种方式，必须要解决资源引用问题。否则前面打出的jar包都是没用的！！

随着Android 7.0的发布，系统对权限控制更加严格，从6.0开始的动态权限，到7.0对应用私有目录访问限制，都可以看到Android系统的隐私保护的力度越来越大。随着安全性的增强，开发者也需要作出跟多的处理适配最新的权限限制。本文主要解决私有目录文件共享的问题，主要方式就是通过FileProvider进行共享。

上面的resourece中的xml对应的文件filepaths就是共享目录配置文件，该文件必须是xml文件，放在项目的xml目录下。

所有的共享目录都需要在<paths>内指定，而且他们是有一定的规范，像<files-path>标签并不是随意定义的xml标签。有一下几种类型的标签：

  • <files-path />

    表示应用内部存储空间中的文件子目录，即：通过Context.getFilesDir()获取到的目录

  • <cache-path />

    表示应用内部的缓存子目录，即：通过getCacheDir()获取到的目录。

  • <external-path />

    表示外部存储子目录，Environment.getExternalStorageDirectory()获取到的目录。

  • <external-files-path />

    表示外部存储目录中的文件子目录，Context#getExternalFilesDir(String)获取的目录。

  • <external-cache-path />

    表示外部存储目录中的缓存子目录，Context.getExternalCacheDir()获取到的目录。

以上的每个标签都有path和name属性，其中path用来指定对应的目录下的子路径，如果path=""则代表使用对应的目录作为根目录，即：如果使用的是<file-paths /> 那么内部的存储空间files就是共享的根目录。

共享文件

共享文件的逻辑还是在应用A中提供，它接受所有android.intent.action.PICK Action的Intent，根据FileProvider生成共享文件的uri，对外提供。

在这里获取共享文件的应用B需要以startActivityForResult（）的方式，使用隐式的Intent去call起应用A,在应用A生成相应的uri后，将结果返回给应用B(setResult())，应用B在onActivityResult()中获取结果。

临时权限设置

文件共享中的关键一步就是共享的应用(应用A)需要临时授权希望获取的应用(应用B)有访问自身私有目录下文件的权限。设置这种权限有以下两种方式：

使用grantUriPermission()

调用Context.grantUriPermission(package, Uri, mode_flags)方法对指定的uri设置授权，mode_flages可以设置为FLAG_GRANT_READ_URI_PERMISSION或 FLAG_GRANT_WRITE_URI_PERMISSION或者两个都设置。但是这种权限是一种持久权限，一旦授予后就会持续有效，直到设备重启或者手动调用revokeUriPermission()方法回收权限。

Intent中添加Flags

使用这种方式需要：1、将共享的uri设置到intent的data(调用intent的setData())；2、为intent添加flags可以为 FLAG_GRANT_READ_URI_PERMISSION or FLAG_GRANT_WRITE_URI_PERMISSIONor 两个一起；3、使用setResult() 将Intent返回。这种方式的授权，权限是在调用共享的Activity的生命周期只能，一旦该Activity出栈了，该权限就失效了。

需要说明的是，在通常的使用中都会选择第二种方式，第一种方式权限的有效期时间过长，虽然可以控制但是需要手动去调用回收权限的操作，这种情况对触发调用的时机也不好把控，第二种授予的权限的有效期随着Activity的生命周期一起结束，这种情况更加满足我们的需求。

总结

随着Android系统的不断升级，对权限限制的更加严格，对于应用间的文件共享，我们为了确保不在最新的系统上出现崩溃的情况，在所有存在文件共享的场景，都要使用FileProvider去实现。其实对于调用端的APP来说，并没有增加工作量，唯一增加的是一个临时权限的设置，另一个是将原来的file://类型替换成content://类型。如果需要做一个对外提供文件共享的应用，那么对于provider的配置也是必不可少的。
